032/**
033 * Finds text that matches a given regular expression. The match is processed by the given {@link MatchProcessor}. The
034 * default match processor replaces the matched texts with the configured replacements.
035 * <p>
036 * <h1>Contents</h1>
037 * <p>
038 * <b> <a href="#g1">1. How do I use this modifier?</a><br/>
039 * <a href="#g2">2. Instead of replacing text I want to do something else when the regular expression matches. How can I
040 * do this?</a><br/>
041 * <a href="#g3">3. How can I find out the position of the matches within the stream?</a> <br/>
042 * <a href="#g4">4. How much memory does the modifier consume?</a><br/>
043 * <a href="#g5">5. Which features of Java's Pattern are not supported yet?</a><br/>
044 * <a href="#g6">6. Which features of Java's Matcher are not supported yet?</a><br/>
045 * <a href="#g7">7. How do I configure RegexModifier if my pattern contains ^ or \b or \B?</a><br/>
046 * <a href="#g8">8. What value to choose for the constructor parameter <code>minimumLengthOfLookBehind</code>?</a><br/>
047 * <a href="#g9">9. What value to choose for the constructor parameter <code>newNumberOfChars</code>?</a><br/>
048 * <a href="#g10">10. Can I share RegexModifier instances between threads?</a><br/>
049 * </b> <!-- ++++++++++++++++++++++++++++++ -->
050 * <p>
051 * <h3 id="g1">1. How do I use this modifier?</h3>
052 * <p>
053 * EXAMPLE:
054 * <code><pre class="prettyprint lang-java">// choose the character stream to modify
055Reader originalReader = new StringReader("edit stream");
056
057// define the regular expression and the replacement
058Modifier myModifier = new RegexModifier("edit stream", 0, "modify stream");
059
060// create the modifying reader that wraps the original reader
061Reader modifyingReader = new ModifyingReader(originalReader, myModifier);
062
063// use the modifying reader instead of the original reader
064String output = IOUtils.toString(modifyingReader);
065assertEquals("modify stream", output);</pre></code>
066 * <h3 id="g2">2. Instead of replacing text I want to do something else when the regular expression matches. How can I
067 * do this?</h3>
068 * <p>
069 * Implement your own {@link MatchProcessor}. In the following example the processor prints the matched text on the
070 * console.
071 * <p>
072 * EXAMPLE: <code><pre class="prettyprint lang-java">class MatchPrinter implements MatchProcessor {
073
074    public MatchProcessorResult process(StringBuilder characterBuffer,
075            int firstModifiableCharacterInBuffer, MatchResult matchResult) {
076
077        // print the matches text
078        System.out.println("match: " + matchResult.group());
079
080        // continue matching behind the end of the matched text
081        return new MatchProcessorResult(matchResult.end(), true);
082    }
083}
084// ...
085Modifier myModifier = new RegexModifier("^.*ERROR.*$", Pattern.MULTILINE, new MatchPrinter(), 0, 2048);</pre></code>
086 * <h3 id="g3">3. How can I find out the position of the matches within the stream?</h3>
087 * <p>
088 * You must count the characters that you skip. You can do this by subclassing {@link RegexModifier} and overwrite
089 * {@link Modifier#modify(StringBuilder, int, boolean)}.
090 * <p>
091 * You might find {@link LineColumnAwareModificationFactory} and {@link PositionAwareModificationFactory} helpful as
092 * well.
093 * <h3 id="g4">4. How much memory does the modifier consume?</h3>
094 * <p>
095 * The maximum buffer size used by this modifier does not exceed the size of the longest match by factor three. EXAMPLE:
096 * Assume in your stream the longest match contains 100K characters. Then the internally used buffer is at no time
097 * larger than 300K characters. But this rule does not apply if you use greedy operators. If you use greedy operators,
098 * the entire stream content might be loaded into the memory at once. The web page of this project gives <a href=
099 * "https://github.com/rwitzel/streamflyer/blob/wiki/AdvancedRegularExpressionsExample.md" >more details</a>.
100 * <h3 id="#g5">5. Which features of Java's Pattern are not supported yet?</h3>
101 * <p>
102 * Apart from \G (the boundary matcher that matches the end of the previous match) all other features mentioned in
103 * {@link Pattern} are supported.
104 * <h3 id="#g6">6. Which features of Java's Matcher are not supported yet?</h3>
105 * <p>
106 * Java's {@link Matcher} allows the user to {@link Matcher#usePattern(Pattern) change} the used pattern. RegexModifier
107 * does not enables this out-of-the-box but {@link StateMachine} provides that functionality.
108 * <p>
109 * Preferences that are region-related (like anchoring bounds, transparent bounds) and preferences that determine how to
110 * continue the matching do not have to be configured because they are already setup in an appropriate way by the
111 * RegexModifier.
112 * <h3 id="#g7">7. How do I configure RegexModifier if my pattern contains ^ or \b or \B?</h3>
113 * <p>
114 * These pattern constructs require a look-behind. Therefore, set <code>minimumLengthOfLookBehind</code> at least to
115 * one. As RegexModifier modifies the input, look-behind constructs may not lead to the expected result (see project
116 * page). Therefore, try to replace the look-behind construct with a normal group, like "\n" for "^" and "\W" for "\b".
117 * <h3 id="#g8">8. What value to choose for the constructor parameter <code>minimumLengthOfLookBehind</code>?</h3>
118 * <p>
119 * If your pattern does not contain zero-width look-behind constructs, then you can choose any value. Zero is
120 * recommended.
121 * <p>
122 * Choose one if your pattern contains zero-width look-behind constructs like ^ or \b or \B but not constructs like
123 * <code>(?&lt;=X)</code> or <code>(?&lt;!X)</code>.
124 * <p>
125 * If your pattern contains zero-width look-behind constructs like <code>(?&lt;=X)</code> or <code>(?&lt;!X)</code>, you
126 * have to find out how many characters the matcher needs at least to match properly by looking behind. Use this number
127 * as value for <code>minimumLengthOfLookBehind</code>. EXAMPLE: Let's say your pattern might check for
128 * <code>(?&lt;=a{3})</code> and <code>(?&lt;=b{5})</code> in front of the actual match. Then the appropriate value is
129 * five. If you set the value to a lower value, then there is no guarantee that the modifier matches properly.
130 * <h3 id="#g9">9. What value to choose for the constructor parameter <code>newNumberOfChars</code>?</h3>
131 * <p>
132 * This value determines how many characters are usually processed at once. If the value is too high or too low, the
133 * performance decreases. The optimal value depends on the length and number of matches in the stream.
134 * <h3 id="#g10">10. Can I share RegexModifier instances between threads?</h3>
135 * <p>
136 * No, a RegexModifier instance is not thread-safe. Use thread-specific instances.
137 * 
138 * @author rwoo
139 * @since 18.06.2011
140 */

143    // * <p>
144    // * <h2>A summary of the internal algorithm (some details are left
145    // out)</h2>
146    // * <p>
147    // * Is there a match in the buffer?
148    // * <ul>
149    // * <li>we found a match
150    // * <ul>
151    // * <li>entire buffer matches and end of stream not hit yet -> the match
152    // might
153    // * change with more input -> FETCH_MORE_INPUT (<i>match_open</i>)
154    // * <li>replace the matched content, and then
155    // * <ul>
156    // * <li>the match processor decides that no skip is needed yet -> continue
157    // with
158    // * the existing buffer content -> try another match
159    // (<i>match_n_continue</i>)
160    // * <li>skip needed but no characters left in the buffer after the
161    // replacement ->
162    // * MODIFY_AGAIN_IMMEDIATELY is only thing we can do
163    // (<i>match_n_refill</i>)
164    // * <li>skip needed and there are characters left in the buffer -> SKIP
165    // * (<i>match_n_skip</i>)
166    // * </ul>
167    // * </ul>
168    // * <li>we haven't found a match.
169    // * <ul>
170    // * <li>By looking for matches (including the empty string) that start in
171    // the
172    // * range [from, maxFrom], the end of the buffer is hit.
173    // * <ul>
174    // * <li>end of stream hit -> no match possible -> SKIP the entire buffer
175    // * (<i>nomatch_eos</i>)
176    // * <li>end of stream not hit -> match might be possible or end of buffer
177    // is hit
178    // * -> FETCH_MORE_INPUT cannot be wrong (<i>nomatch_fetch</i>)
179    // * </ul>
180    // * <li>We did not match a single character or the empty string -> SKIP the
181    // * entire buffer (<i>nomatch_skip</i>)
182    // * </ul>
183    // * </ul>

310    /**
311     * @see com.github.rwitzel.streamflyer.core.Modifier#modify(java.lang.StringBuilder, int, boolean)
312     */
313    @Override
314    public AfterModification modify(StringBuilder characterBuffer, int firstModifiableCharacterInBuffer,
315            boolean endOfStreamHit) {
316
317        // the first position we will match from.
318        Integer minFrom = null;
319
320        while (true) {
321
322            // determine the range [minFrom, maxFrom] that will contain the
323            // first character of the matching string
324
325            if (minFrom == null) {
326                minFrom = firstModifiableCharacterInBuffer;
327
328                if (unseenCharactersToSkip > 0) {
329
330                    // is there at least one modifiable character in the buffer?
331                    if (minFrom + unseenCharactersToSkip > characterBuffer.length()) {
332                        // no -> we need more input to skip the characters
333
334                        if (endOfStreamHit) {
335                            // -> stop
336                            return factory.stop(characterBuffer, firstModifiableCharacterInBuffer, endOfStreamHit);
337                        } else {
338                            // -> fetch more input
339                            return factory.fetchMoreInput(0, characterBuffer, firstModifiableCharacterInBuffer,
340                                    endOfStreamHit);
341                        }
342
343                    } else {
344                        // yes -> increase the *minFrom*
345                        minFrom += unseenCharactersToSkip;
346                        unseenCharactersToSkip = 0;
347                    }
348                }
349            }
350
351            // we have to restrict maxFrom in order to prevent that the
352            // requested number of characters increases more and more
353            int maxFrom = firstModifiableCharacterInBuffer + newNumberOfChars;
354
355            // adjust maxFrom if it is bigger than the given buffer
356            if (maxFrom > characterBuffer.length()) {
357                // this is NOT set to characterBuffer.length() -1 by intention
358                // as a regular expression might match on the zero length string
359                // (but with positive look-behind)!
360                maxFrom = characterBuffer.length();
361            }
362
363            // find first match
364            // (as the match processor might have modified the buffer, we reset
365            // the matcher inside the loop instead of outside of the loop)
366            matcher.reset(characterBuffer);
367            boolean matchFound = matcher.findUnlessHitEnd(minFrom, maxFrom);
368
369            if (matchFound) {
370                // we found a match
371
372                // could change this positive match into a negative one
373                // (matcher.requireEnd()) or into a longer one (greedy
374                // operator)?
375                if (matcher.hitEnd() && !endOfStreamHit) {
376                    // (match_open) yes, it could -> we need more input
377
378                    int numberOfCharactersToSkip = matcher.lastFrom() - firstModifiableCharacterInBuffer;
379
380                    // read more input (skip some characters if possible)
381                    AfterModification mod = factory.fetchMoreInput(numberOfCharactersToSkip, characterBuffer,
382                            firstModifiableCharacterInBuffer, endOfStreamHit);
383
384                    assert __checkpoint( //
385                            "name", "match_open", //
386                            "minLen", firstModifiableCharacterInBuffer, //
387                            "characterBuffer", characterBuffer, //
388                            "endOfStreamHit", endOfStreamHit, //
389                            "afterModification", mod);
390
391                    return mod;
392                } else {
393                    // no -> thus we can use this match -> process the match
394
395                    // process the match
396                    MatchResult matchResult = matcher; // .toMatchResult()?
397                    // (I could pass firstModifiableCharacterInBuffer instead of
398                    // minFrom as well)
399                    MatchProcessorResult matchProcessorResult = matchProcessor.process(characterBuffer, minFrom,
400                            matchResult);
401                    minFrom = matchProcessorResult.getFirstModifiableCharacterInBuffer();
402
403                    // match again without skip? (even for minFrom == maxFrom we
404                    // try a match) (minFrom <= maxFrom is needed so that the
405                    // buffer does not increase if the replacement is longer
406                    // than the replaced string, i.e. minFrom > maxFrom means
407                    // that a SKIP is needed)
408                    // (I (rwoo) think an earlier SKIP (minFrom < maxFrom
409                    // instead of minFrom <= maxFrom) would also be possible.
410                    // This has no impact on the matching and only minimal
411                    // impact on the performance)
412                    if (minFrom <= maxFrom && matchProcessorResult.isContinueMatching()) {
413                        // (match_n_continue) no skip needed yet -> continue
414                        // matching on the existing buffer content
415
416                        assert __checkpoint( //
417                                "name", "match_n_continue", //
418                                "minLen", firstModifiableCharacterInBuffer, //
419                                "characterBuffer", characterBuffer, //
420                                "endOfStreamHit", endOfStreamHit, //
421                                "minFrom", minFrom);
422
423                        // We try the next match on the modified input, i.e.
424                        // not match only once -> next loop
425                        continue;
426                    } else {
427
428                        // we shall not continue matching on the
429                        // existing buffer content but skip (keep the buffer
430                        // small)
431
432                        int numberOfCharactersToSkip;
433                        if (minFrom > characterBuffer.length()) {
434                            // this happens when we avoid endless loops after
435                            // we matched an empty string
436                            unseenCharactersToSkip = minFrom - characterBuffer.length();
437                            ZzzAssert.isTrue(unseenCharactersToSkip == 1, "unseenCharactersToSkip must be one but was "
438                                    + unseenCharactersToSkip);
439                            numberOfCharactersToSkip = characterBuffer.length() - firstModifiableCharacterInBuffer;
440                        } else {
441                            numberOfCharactersToSkip = minFrom - firstModifiableCharacterInBuffer;
442                        }
443
444                        if (numberOfCharactersToSkip == 0) {
445
446                            // (match_n_refill) there are no characters left in
447                            // the buffer after the replacement ->
448                            // MODIFY_AGAIN_IMMEDIATELY is only thing we can do
449                            // (the match processor implementation must not
450                            // cause an endless loop)
451
452                            // (passing false for endOfStreamHit is ugly!!!)
453                            // we should offer a new method in
454                            // ModificationFactory, something like
455                            // continueAfterModification(...) that chooses the
456                            // appropriate action. the following code is always
457                            // a MODIFY_AGAIN_IMMEDIATELY
458                            AfterModification mod = factory.fetchMoreInput(numberOfCharactersToSkip, characterBuffer,
459                                    firstModifiableCharacterInBuffer, false);
460
461                            assert __checkpoint( //
462                                    "name", "match_n_refill", //
463                                    "minLen", firstModifiableCharacterInBuffer, //
464                                    "characterBuffer", characterBuffer, //
465                                    "endOfStreamHit", endOfStreamHit, //
466                                    "afterModification", mod);
467
468                            return mod;
469                        } else {
470                            // (match_n_skip) there are characters left in
471                            // the buffer -> SKIP
472
473                            AfterModification mod = factory.skipOrStop(numberOfCharactersToSkip, characterBuffer,
474                                    firstModifiableCharacterInBuffer, endOfStreamHit);
475
476                            assert __checkpoint( //
477                                    "name", "match_n_skip", //
478                                    "minLen", firstModifiableCharacterInBuffer, //
479                                    "characterBuffer", characterBuffer, //
480                                    "endOfStreamHit", endOfStreamHit, //
481                                    "afterModification", mod);
482
483                            return mod;
484                        }
485                    }
486
487                }
488            } else {
489                // we haven't found a match
490
491                // By looking for matches (including the empty string) that
492                // start in the range [from, maxFrom], is the end of the buffer
493                // hit?
494                if (matcher.lastFrom() <= maxFrom) {
495                    // yes, the end of the buffer was hit
496
497                    // can we get more input?
498                    if (endOfStreamHit) {
499                        // (nomatch_eos) no, in the entire stream we will not
500                        // found more matches that start in range [from,
501                        // maxFrom] -> skip the characters from range [from,
502                        // maxFrom]
503
504                        int numberOfCharactersToSkip = maxFrom - firstModifiableCharacterInBuffer;
505                        AfterModification mod = factory.skipOrStop(numberOfCharactersToSkip, characterBuffer,
506                                firstModifiableCharacterInBuffer, endOfStreamHit);
507
508                        assert __checkpoint( //
509                                "name", "nomatch_eos", //
510                                "minLen", firstModifiableCharacterInBuffer, //
511                                "characterBuffer", characterBuffer, //
512                                "endOfStreamHit", endOfStreamHit, //
513                                "afterModification", mod);
514
515                        return mod;
516
517                    } else {
518                        // (nomatch_fetch) yes > we should fetch more input
519                        // (because end of stream is not hit yet)
520
521                        // if maxFrom == characterBuffer.length() and lastFrom()
522                        // == maxFrom we cannot decide whether this is really an
523                        // open match or rather a not a match at all. But by
524                        // skipping the characters in front of lastFrom() and
525                        // fetching more input we cannot do anything wrong
526
527                        int numberOfCharactersToSkip = matcher.lastFrom() - firstModifiableCharacterInBuffer;
528                        AfterModification mod = factory.fetchMoreInput(numberOfCharactersToSkip, characterBuffer,
529                                firstModifiableCharacterInBuffer, endOfStreamHit);
530
531                        assert __checkpoint( //
532                                "name", "nomatch_fetch", //
533                                "minLen", firstModifiableCharacterInBuffer, //
534                                "characterBuffer", characterBuffer, //
535                                "endOfStreamHit", endOfStreamHit, //
536                                "afterModification", mod);
537
538                        return mod;
539                    }
540
541                } else { // matcher.lastFrom() == maxFrom + 1
542
543                    // (nomatch_skip) no, we are matching not a single character
544
545                    // -> skip the characters from range [from, maxFrom]
546                    int numberOfCharactersToSkip = maxFrom - firstModifiableCharacterInBuffer;
547                    AfterModification mod = factory.skipOrStop(numberOfCharactersToSkip, characterBuffer,
548                            firstModifiableCharacterInBuffer, endOfStreamHit);
549
550                    assert __checkpoint( //
551                            "name", "nomatch_skip", //
552                            "minLen", firstModifiableCharacterInBuffer, //
553                            "characterBuffer", characterBuffer, //
554                            "endOfStreamHit", endOfStreamHit, //
555                            "afterModification", mod);
556
557                    return mod;
558                }
559            }
560        }
561
562    }
